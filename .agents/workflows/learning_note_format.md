# 📚 プログラミング学習メモ フォーマット規約

コードの悪い例（アンチパターン）と良い例（改善版）を比較して学習ノートを作成する際は、必ず以下のフォーマット（雛形）に従ってMarkdownを作成すること。
また、**太字（ボールド）** を使用する際は、必ず前後に半角スペースを入れること（例: `文字 **「太字」** 文字`）。
Pythonのコードブロックでは、必ず **型ヒント（Type Hints）** を記載すること（引数の型、戻り値の型 `-> None` / `-> int` 等、フィールドの型注釈）。
専門用語（カプセル化、不変条件、DRY原則 など）が登場する箇所には、必ず **💡 用語Tips** を引用ブロックで追記し、初学者が読んでもすぐ理解できるよう1〜2行で噛み砕いた解説を記載すること。形式は以下の通り：
`> 💡 **用語: [用語名]（[英語名]）とは？**`
`> [初学者向けの簡潔な解説]`

## ⚠️ 正確性に関するルール

学習ノートの記述が **誤解を招く表現や不正確な断言** にならないよう、以下のルールを厳守すること。

1. **「〜は常にダメ」「〜は必ずアンチパターン」のような絶対的な断言を避ける**
   * 技術的な手法には大抵「使うべき場面」と「使うべきでない場面」がある。どちらか一方だけを強調すると誤解を招く。
   * ⭕️ 「〜の場合は問題になる」「〜のケースでは避けるべき」のように **条件付きで** 記載する。
2. **各言語の仕様の違いを正確に記載する**
   * 例: Python には Java のような厳密な `private` は存在しない（`_` は慣習）。このような差異を省略せず記載する。
3. **わからない場合は公式ドキュメントやリファレンスを確認する**
   * 用語の定義や言語仕様について不確かな場合は、公式ドキュメント（Oracle JavaDoc、Python PEP、MDN Web Docs 等）を参照してから記載する。
   * 不確かな情報を推測で書いてはならない。

## 適用フォーマット（テンプレート）

```markdown
## [テーマ名（例: 技術駆動命名）]

**[言語名（例: Java）]:**
\`\`\`[言語の短縮名]
[悪いコードの例]
\`\`\`

**[その他の言語名（Python）]:**
\`\`\`[言語の短縮名]
[悪いコードの例]
\`\`\`


**[その他の言語名（typescript）]:**
\`\`\`[言語の短縮名]
[悪いコードの例]
\`\`\`


---

### 🚨 何がダメなのか？ ([テーマ名]の問題点)

記載されているコードには、以下の問題があります。

#### 1. [問題点のタイトル]
> 対象: \`[問題となっているコード箇所や変数名など]\`

[問題点の詳細な解説。なぜ良くないのか、どのような弊害があるのかを具体的に記載する。]

#### 2. [問題点のタイトル]
> 対象: \`[対象]\`

[解説...]

---

### 🌟 改善版コード（[改善のアプローチ名]）

[どのように改善するのかの全体的な方針やアプローチを簡潔に記載]


**[言語名]:**
\`\`\`[言語の短縮名]
[改善されたコードの例]
\`\`\`

#### 改善のポイント
- \`[悪い例]\` → **\`[改善例]\`**: [なぜそのように変更したのか、どう改善されたのかの理由]
- ...

#### 📝 外部からの使い方（使用例） ※クラスを用いた改善版の場合

[改善版のクラスを外部から実際に使うコード例を記載する。インスタンス生成、メソッド呼び出し、結果の取得までの流れを具体的に示す。]

---

### 🎯 まとめ（[テーマ]における考え方）

[今回のテーマ]から脱却し、正しいコードを書くための考え方を「目的・目標・手段」の順に整理します。

#### 1. 目的（Why: なぜこれにこだわるのか？）
* **「[目的のキャッチコピー]」**
  * [目的の詳細な理由1]
  * [目的の詳細な理由2]

#### 2. 目標（What: どのような状態を目指すのか？）
* **「[目標のキャッチコピー]」**
  * [目指すべき状態の具体的な説明]

#### 3. 手段（How: どうやって達成するのか？）
* **「[手段のキャッチコピー]」**
  * ❌ \`[悪い例]\` → ⭕️ \`[良い例]\` ([簡単な解説])
  * [具体的なアクションや意識すべきこと]
```
